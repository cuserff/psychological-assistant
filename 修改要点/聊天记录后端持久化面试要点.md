# 聊天记录后端持久化 —— 面试要点

## 一、为什么要做这个功能？

项目最初的聊天记录**仅存储在浏览器 localStorage** 中，存在三个致命问题：
1. 用户清除浏览器缓存，所有对话记录丢失
2. 换一台设备或浏览器登录，看不到任何历史对话
3. localStorage 有 5MB 容量上限，对话多了会溢出

因此需要将聊天会话持久化到后端服务器，实现**跨设备、跨浏览器的数据持久保存**。

## 二、技术方案设计

### 存储方案选择

| 方案 | 优点 | 缺点 | 是否采用 |
|------|------|------|----------|
| 单个 JSON 文件存所有用户 | 实现简单 | 并发读写冲突，文件过大 | 否 |
| **每用户独立 JSON 文件** | 隔离性好，无并发冲突 | 用户量大时文件数多 | **是** |
| 数据库（SQLite/MySQL） | 查询灵活，性能好 | 引入额外依赖，复杂度高 | 后续升级 |

最终采用 **`backend/data/chats/{userId}.json`**，每个用户一个独立文件，与项目现有的 `users.json` 文件存储模式保持一致。

### API 设计

| 端点 | 方法 | 鉴权 | 说明 |
|------|------|------|------|
| `/api/chat/sessions` | GET | JWT | 获取当前用户的所有会话 |
| `/api/chat/sessions` | PUT | JWT | 全量覆盖保存所有会话 |
| `/api/chat/sessions/:sessionId` | DELETE | JWT | 删除指定会话 |

**为什么用 PUT 全量覆盖而不是增量更新？**
- 前端 store 中 sessions 数组是整体管理的（创建、排序、消息追加都在前端完成）
- 全量覆盖避免了前后端数据diff和冲突合并的复杂度
- 对于单用户场景，JSON 文件的全量写入性能完全够用

## 三、核心实现

### 1. 后端：数据读写 + API 端点

```javascript
// 读取用户聊天记录，文件不存在返回空数组
function readUserChats(userId) {
  const filePath = path.join(CHATS_DIR, `${userId}.json`)
  if (!fs.existsSync(filePath)) return []
  return JSON.parse(fs.readFileSync(filePath, 'utf-8'))
}

// 写入用户聊天记录
function writeUserChats(userId, sessions) {
  const filePath = path.join(CHATS_DIR, `${userId}.json`)
  fs.writeFileSync(filePath, JSON.stringify(sessions, null, 2), 'utf-8')
}
```

所有接口都经过 `authMiddleware` 鉴权，通过 JWT token 中的 `req.user.id` 确保**用户只能访问自己的数据**。

### 2. 前端 API 层：封装三个接口调用

```javascript
// 复用项目已有的 request() 封装，自动注入 Authorization 头
export function fetchSessions() {
  return request('/api/chat/sessions')
}

export function saveSessionsToServer(sessions) {
  return request('/api/chat/sessions', {
    method: 'PUT',
    body: JSON.stringify({ sessions })
  })
}

export function deleteSessionFromServer(sessionId) {
  return request(`/api/chat/sessions/${sessionId}`, {
    method: 'DELETE'
  })
}
```

### 3. chatStore 改造：双写策略

**loadSessions（加载）—— 后端优先，localStorage 兜底：**

```javascript
async function loadSessions() {
  try {
    // 优先从后端加载
    const res = await apiFetchSessions()
    sessions.value = res.data
    // 同步写入 localStorage 作为缓存
    setItem(key, res.data)
  } catch {
    // 后端不可用时，降级到 localStorage
    const saved = getItem(key)
    sessions.value = saved || []
  }
}
```

**saveSessions（保存）—— 双写，localStorage 即时 + 后端异步：**

```javascript
function saveSessions() {
  // localStorage 立即写入 → 保证 UI 响应速度
  setItem(key, sessions.value)
  // 异步同步到后端 → 不阻塞用户操作
  saveSessionsToServer(sessions.value).catch(() => {})
}
```

## 四、架构设计亮点

### 双写缓存策略
```
用户操作 → saveSessions()
             ├── localStorage.setItem()    ← 同步，即时生效
             └── PUT /api/chat/sessions    ← 异步，不阻塞 UI

页面加载 → loadSessions()
             ├── GET /api/chat/sessions    ← 优先后端（数据源）
             │     └── 成功 → 回写 localStorage（刷新缓存）
             └── fallback → localStorage   ← 后端不可用时降级
```

这种策略的好处：
- **即时性**：localStorage 同步写入，用户无感知延迟
- **可靠性**：后端作为持久化数据源，跨设备可访问
- **容错性**：后端挂了不影响当前设备的使用体验

### 最小改动原则
- 后端保持了与 `users.json` 一致的文件存储模式，没有引入新依赖
- 前端 store 的对外接口（`loadSessions`、`saveSessions`）签名不变，其他组件零改动
- SSE 流式对话逻辑完全不受影响，`finalizeReply()` 内部调用 `saveSessions()` 自动触发后端同步

## 五、安全考量

1. **数据隔离**：通过 JWT 中的 userId 作为文件名，用户只能读写自己的聊天文件
2. **鉴权保护**：所有接口都经过 `authMiddleware`，未登录或 token 过期返回 401
3. **输入校验**：PUT 接口校验 sessions 必须是数组，防止非法数据写入

## 六、面试话术示例
